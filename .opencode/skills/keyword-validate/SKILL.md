# Validate Keyword

## 目标

这个 skill 不是判断“这个词有没有量”，而是判断：

- 这个词背后是不是一个真实需求
- 这个需求是不是能做成页面、工具或站点
- 这个词是不是能扩成词簇，而不是单点词
- 这个词对应的 SERP 是否给独立开发者留了口子
- 这个词背后是否有明确商业价值，而不只是空流量

最终结论必须同时回答两件事：

1. 值不值得做
2. 值得做的话，应该做成什么

## 必用技能和工具

默认至少组合使用这些来源，不要只靠一个信号下判断：

1. `use-google-trends`
2. `serp-inspection`
3. `x-search`
4. `web-access`
5. `use-semrush` 或 `seo` MCP tools

补充来源：

- `use-webcafe`：当你对这个需求的产品形态、切入策略、独立开发者打法不够确定时，用它补方法论和案例，不要只补表面观点。

## 适用场景

用于这类任务：

- “这个 keyword 值得做吗？”
- “这个词有商业价值吗？”
- “这个搜索词能不能做成网站 / 工具 / SaaS / 落地页？”
- “这个词该不该投入时间做 SEO？”

不适用于：

- 纯找一批新词
- 纯做站点采样
- 纯做 backlink 研究

这些场景优先用 `keyword-research`。

## 核心原则

### 1. 不从单词直接下结论

如果用户只给了一个词，也不要只看这个词本身。

先把它扩成一个小词簇再判断：

- 近义表达
- 更具体的长尾表达
- 错拼或口语表达
- 相邻任务型词
- 带商业意图修饰词的变体

如果一个词根本扩不出自然词簇，它的价值默认下降，除非单词本身极强。

### 2. 先判断需求，再判断流量

“有流量”不等于“值得做”。

先判断：

- 搜的人到底想完成什么任务
- 这个任务是不是反复出现
- 这个任务是不是有页面承接形态
- 用户有没有付费、注册、提交线索、下载、安装或比较工具的倾向

### 3. 商业价值必须单独判断

不要把“值得做 SEO”直接等同于“有商业价值”。

商业价值至少看这几类信号：

- 搜索词是否带有明显交易或工具意图
- 现有排名页是否在卖产品、收线索、做 affiliate、推订阅或转化安装
- 用户痛点是否强到愿意付费，或至少愿意留下高价值行为
- 这个需求是否靠近预算、效率、收入、风险、合规、工作结果这类高价值场景
- 是否可以自然接到 SaaS、工具、模板库、服务线索、数据订阅或 affiliate

### 4. 最终判断看“页面机会”而不是抽象关键词

最后要回答的不是“词好不好”，而是：

- 应该做工具页
- 应该做模板页
- 应该做比较页
- 应该做目录页
- 应该做程序化页面集群
- 还是根本不该做

### 5. 公式只做辅助，不替代真人判断

笔记里已经验证过，以下公式有用，但都不能越过 `SERP 意图判断`：

- `KDRoi = (搜索量 × CPC) / KD`
- `KGR = intitle / 搜索量`
- `KD` 可近似映射到所需外链预算，但只是估算，不是最终裁决

顺序必须是：

1. 先看 `SERP` 意图是否对
2. 再看 `KD / 搜索量 / CPC`
3. 再看 `Trends`
4. 最后结合 `KGR / intitle` 和外链预算做决策

如果 `SERP` 意图不对，其他公式一律降为无效参考。

## 可用公式

### 1. `KDRoi`：商业价值辅助排序

公式：

```text
KDRoi = (搜索量 × CPC) / KD
```

用途：

- 用来粗排哪些词“离钱更近”
- 适合在一组候选词里做相对比较

解释：

- 搜索量代表需求大小
- `CPC` 代表商业价值强弱
- `KD` 代表优化成本

注意：

- 这是排序公式，不是生死公式
- `KDRoi` 高，但 `SERP` 被品牌和巨头锁死，照样不做
- 没有 `CPC` 时，不要硬算，改用商业意图人工判断

### 2. `KGR`：页面竞争密度粗筛

公式：

```text
KGR = intitle / 搜索量
```

其中：

- `intitle` = Google 搜 `intitle:"关键词"` 的结果数

经验阈值：

- `KGR < 0.25`：强烈推荐
- `KGR 0.25 - 1.0`：一般，需要结合 `SERP` 和外链再看
- `KGR > 1.0`：偏拥挤，新站慎选

注意：

- `KGR` 只用于粗筛页面竞争密度
- 不能替代 `SERP` 检查
- `KGR` 低但 `SERP` 第一名是 `NYT / App Store / Steam / Amazon` 这类巨头时，依然不值得做

### 3. `KD` 到外链预算的估算

Web.Cafe 笔记中的经验拟合公式：

```text
y = (-89.32212 × x) / (-100.6193 + x)
```

其中：

- `x = KD`
- `y = 所需外链数量（估算）`

速查表：

- `KD 10` -> 约 `9` 个外链
- `KD 20` -> 约 `20` 个外链
- `KD 30` -> 约 `33` 个外链
- `KD 40` -> 约 `47` 个外链
- `KD 50` -> 约 `63` 个外链
- `KD 60` -> 约 `74` 个外链
- `KD 70` -> 约 `85` 个外链

正确用法：

- 把 `KD` 当预算提示，不当最终结论
- 再去看前排站点真实外链域名数、流量和页面质量
- 如果别人外链投入很大但流量不高，说明 ROI 差
- 如果前排站外链不多、页面又弱，是积极信号

## 默认流程

### 1. 定义这个词的真实任务

先把 keyword 翻成用户任务，而不是停留在字面。

至少写清：

- 用户可能想完成什么事
- 这更像工具需求、内容需求、比较需求还是导航需求
- 一个人会不会反复搜索或持续遇到这个问题

### 2. 扩成小词簇

优先用：

- `seo_keyword_generator`
- `seo_keyword_difficulty`
- `use-google-trends`

至少拿到：

- 主词
- 2-5 个近义或相邻变体
- 大致 KD / 难度感知
- 哪种表达更接近真实用户语言

如果你发现这个词只能长成几个机械变体，不能长成自然页面群，要明确降级。

### 3. 用 Trends 判断需求强弱和表达自然度

使用 `use-google-trends` 比较主词和相邻表达。

重点判断：

- 过去 12 个月是上升、平稳还是下滑
- 哪种表达是主流表达
- 是长期需求，还是热点脉冲
- 是否出现新的 rising queries，说明需求在扩展

规则：

- Trends 只做排序和语言验证
- 不要把 Trends 当绝对搜索量
- 趋势很好但意图脏，也不能直接判值得做

### 3.5 进入验证池后，先补关键字段

在正式给结论前，默认把候选词补成一个小验证池。至少补这些字段：

- `Trend`
- `KD`
- `CPC`（如果拿得到）
- `intitle`
- `KGR`
- `SERP` 观察

不要一边扩词一边冲动决定上线。

### 4. 人工看 SERP，判断能不能打

使用 `serp-inspection`，并按其要求走 `web-access`。

至少判断：

- Google 当前奖励的主意图是什么
- 前 5 个自然结果主要是什么页面类型
- 前排是强品牌强页面，还是一堆软柿子
- 结果里有没有论坛、低质量 listicle、老旧工具、意图不准的页面
- 你能不能做出一个明显更匹配的页面或工具

如果 SERP 前排已经被高质量强品牌 exact-match 页面锁死，哪怕趋势和 KD 不错，也要降级。

### 5. 看社区语言，确认痛点是否真实

使用 `x-search`。

重点找：

- 用户是否在重复抱怨这个问题
- 用户原话里是怎么描述这个需求的
- 有没有替代方案不满意、求推荐、求工具、求模板、求比较
- 是真实问题，还是只有营销人在谈

如果 X 信号不够，再用 `web-access` 看：

- Reddit
- GitHub issues / discussions
- Product Hunt
- Chrome Web Store reviews
- 竞品评论区
- niche forums

### 6. 单独检查商业化路径

这个步骤不能省。

使用 `web-access` 打开排名页、竞品站、相关工具页，判断：

- 这些页面怎么赚钱
- 是广告流量型、affiliate、卖工具、卖模板、卖 leads，还是纯内容
- 用户完成一次搜索后，最自然的下一步动作是什么
- 这个动作是否接近转化

高商业价值常见信号：

- 用户在找具体工具、服务、模板、软件、替代品、对比、价格、最佳方案
- 排名页已经大量出现产品页、比较页、登录页前的营销页、pricing 相邻页
- 问题足够频繁或足够痛，能支持订阅、按次收费或高价值 lead

低商业价值常见信号：

- 需求只是一次性好奇
- 搜索者更像想看定义、新闻或八卦
- 前排主要是纯资讯页，且没有后续转化动作
- 即使拿到流量，也很难接产品或商业闭环

### 7. 必要时用 Web.Cafe 校正打法判断

当你对“这个词该做不该做”没有把握时，用 `use-webcafe` 搜这类主题：

- 挖需求
- niche validation
- SEO
- tools
- SaaS
- affiliate

重点不是找结论，而是校正这几个判断：

- 独立开发者是否真的适合做这种机会
- 这种需求更适合内容站、工具站还是产品站
- 是适合 SEO 切入，还是应该先靠分发、社区或 cold start

如果 Web.Cafe 只能提供泛泛观点，不要让它推翻已有硬证据。

## 评分框架

对每个 keyword，至少给这 5 项判断：

1. 需求强度
2. 意图清晰度
3. SERP 可打性
4. 词簇扩展性
5. 商业价值

可用 `高 / 中 / 低`，也可用 `1-5`，但最终必须落到明确结论。

如果需要更快的统一判断，可以加一层辅助打分：

```text
Worth Doing Score =
  Intent Clarity +
  SERP Beatable +
  Cluster Expansion +
  Commercial Value +
  Trend Health
  - Brand/IP Risk
  - ROI Mismatch
```

解释：

- `Intent Clarity`：Google 是否明确把它当成你要做的需求
- `SERP Beatable`：前排是否有软柿子页面
- `Cluster Expansion`：能否自然长出 `3-10` 个相关页
- `Commercial Value`：能否接产品、订阅、lead、affiliate 或高价值动作
- `Trend Health`：趋势是活的，不是纯脉冲
- `Brand/IP Risk`：品牌词、商标词、IP 风险词要直接扣分
- `ROI Mismatch`：需要大量外链预算但商业回报弱，也要重扣

这个分数只用于帮助你组织判断，不要伪装成精确数学模型。

## 最终结论类型

结论只允许这四类：

1. **值得做**
   - 需求真实
   - 意图清楚
   - SERP 有机会
   - 能接商业化

2. **值得做，但只能用细分角度切入**
   - 主词太硬
   - 但某个 niche angle、页面类型或长尾集群还能打

3. **可以观察，不建议现在投入**
   - 有一些信号
   - 但趋势、SERP、商业化或需求强度还不够扎实

4. **不值得做**
   - 即使有量，也不建议投入

## 输出要求

默认输出必须包含：

1. `Keyword`
2. `Core task behind the keyword`
3. `Keyword cluster checked`
4. `Trend summary`
5. `SERP summary`
6. `Community signal summary`
7. `Commercial value`
8. `Best page / product angle`
9. `Final verdict`
10. `Formula check`（`KDRoi / KGR / KD budget hint`）

推荐输出格式：

```markdown
## Keyword Validation

- Keyword: ...
- Core task: ...
- Verdict: 值得做 / 值得做，但只能用细分角度切入 / 可以观察，不建议现在投入 / 不值得做
- Commercial value: 高 / 中 / 低

### Why

- Trend: ...
- SERP: ...
- Community: ...
- Monetization: ...
- Cluster expansion: ...

### Best angle

- Recommended page type: ...
- First MVP page: ...
- Expansion path: ...
```

## 硬性避免事项

- 不要只看一个 SERP 截图就下结论
- 不要只凭 KD 或 Trends 说“值得做”
- 不要把“有搜索量”误判为“有商业价值”
- 不要忽略词簇扩展性
- 不要只说能做，必须说应该做成什么
- 不要在没有看商业化路径的情况下说“这个词能赚钱”
