# AWS 计算抽象与扩缩容取舍

## 定义

这个概念页总结一种面向 AWS 架构和面试表达的核心心智模型：把 `EC2`、`ECS/Fargate`、`Lambda` 看作同一条计算抽象阶梯上的三个层级。层级越高，基础设施责任越少，但运行时控制权和可塑性也越低。

## 核心模型

- `EC2`：持久机器（persistent machines）
- `ECS/Fargate`：持久服务或任务（persistent services/tasks）
- `Lambda`：短暂执行（ephemeral executions）

这条阶梯同时意味着三件事在同步变化：

1. 你要负责的基础设施越来越少。
2. 你能直接控制的运行环境越来越少。
3. 你思考扩缩容的单位发生变化。

## 三层的思考方式

### EC2

- 思考问题：`我需要多少台机器？`
- 适合：长时间运行服务、强定制环境、状态型负载、重计算、遗留系统。
- 优势：可 SSH、可调 OS 和网络、没有 cold start、持续负载下成本可能更稳。
- 代价：补丁、部署、扩容、故障恢复、安全更新都要自己承担。

### ECS 或 Fargate

- 思考问题：`我需要多少个容器/任务副本？`
- 适合：持续运行 API、多服务拆分、希望标准化部署和更细粒度扩容的场景。
- 优势：比直接管机器更容易做调度、健康检查、滚动发布和服务级扩缩容。
- 关键区别：
  - `ECS on EC2` 仍然受底层 EC2 容量约束。
  - `ECS on Fargate` 由 AWS 负责底层容量抽象，容量规划负担更低。

### Lambda

- 思考问题：`什么事件发生时，应该执行哪个函数？`
- 适合：事件驱动、异步任务、流量突发、MVP、轻量 API。
- 优势：几乎不用管理服务器，自动扩展很强，低到中等流量下通常高效。
- 代价：cold start、超时和资源限制、状态不能放本地、分布式排障更碎片化。

## 扩缩容的真实差异

### 扩 EC2

- 单位是机器。
- 典型路径是 `ALB + Auto Scaling Group`。
- 核心问题是新机器启动慢、初始化慢，以及应用能否无状态横向扩展。

### 扩 ECS

- 单位是 task / service replica。
- 逻辑上更细粒度，但如果运行在 `EC2` 上，最终仍要看底层机器是否装得下这些 task。
- 常见生产问题是 task 卡在 `PENDING`，原因不是 ECS 不想扩，而是底层 CPU、内存或 placement constraint 不够。

### 扩 Lambda

- 单位接近并发执行数。
- 从使用者视角，不需要先规划机器或集群容量。
- 但这种“容量抽象消失”是以运行时限制和更低控制权换来的。

## 面试里真正想听到的点

1. 你会按工作负载选抽象层，而不是默认某个服务“更先进”。
2. 你知道更高抽象不等于无代价，只是把问题从运维转移到限制管理。
3. 你明白无状态设计是横向扩展的前提，尤其是 EC2/ECS 场景。
4. 你能说清 `ECS 负责调度，EC2 提供容量，Fargate 抽象容量` 这类层次关系。

## 实用表达

可以用一句话概括：

`EC2 提供持久机器，ECS 提供持久服务，Lambda 提供按事件出现的短暂执行。`

这句话适合用来快速建立后续取舍讨论的框架。
