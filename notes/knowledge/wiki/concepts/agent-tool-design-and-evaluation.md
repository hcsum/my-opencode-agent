# Agent Tool Design And Evaluation

## 定义

面向 LLM agent 的工具设计方法：不是把现有 API 薄封装后直接暴露给模型，而是围绕真实任务流、上下文成本、可恢复性与评测反馈来持续打磨工具接口。

## 核心判断

- 工具首先是模型可用性的设计问题，其次才是 API 完整性问题。
- 如果一个任务中的确定性步骤可以由工具内部完成，就不应把这些中间协调负担留给 agent。
- 工具质量不能靠抽象争论判断，必须通过真实任务评测和调用转录来验证。

## 设计原则

### 1. 以工作流为单位设计工具

- 优先提供贴近任务的高杠杆工具，而不是为每个底层接口单独造一个工具。
- 好工具应减少模型在多步检索、拼装参数、筛选无关结果上的 token 消耗。

### 2. 命名空间要清晰

- 当 agent 同时看到多个服务的工具时，命名重叠会显著增加误用概率。
- 前缀式还是后缀式命名不应想当然决定，应该通过评测观察不同模型的选择表现。

### 3. 返回高信号上下文

- 输出应优先保留能支持后续推理的字段，而不是机械返回所有原始字段。
- 对模型更友好的自然语言标识通常优于晦涩技术 ID。
- 如果后续调用必须依赖技术 ID，可以用 `concise` / `detailed` 一类格式选项平衡可读性与可操作性。

### 4. 控制 token 成本

- 通过分页、过滤、范围选择和默认截断来抑制无效上下文膨胀。
- 截断必须显式告知模型，并提示下一步应如何缩小范围。
- 错误信息要可执行，避免只返回难以利用的底层异常。

### 5. 把工具描述当作 prompt 资产

- 描述文本和 schema 不是文档附属物，而是模型决策上下文的一部分。
- 参数名、字段解释、隐含业务规则都应尽量明确。
- 每次描述层改动都值得配合评测验证，因为微小改写也可能改变调用行为。

## 评测方法

### 任务设计

- 评测任务应尽量接近真实工作，而不是单步玩具题。
- 更强的任务通常要求多次工具调用和跨系统整合。
- 验证器要面向结果正确性，避免因无关格式差异过度拒绝。

### 观测指标

- 准确率
- 工具运行时长
- 工具调用次数
- Token 使用量
- 工具报错情况

这些指标可以暴露重复调用、参数选择差、返回结构噪声过多或描述不清等问题。

## 迭代闭环

1. 先做可运行原型。
2. 用真实任务集跑评测。
3. 读取 agent 调用转录定位失败模式。
4. 修改工具实现、描述、schema 或输出格式。
5. 用保留测试集复测，防止过拟合。

## 适用启发

- 设计 MCP 工具时，应优先思考 agent 典型任务，而不是先列出后端已有接口。
- 构建内部工具平台时，评测框架和 transcript review 应与工具实现同等重要。
- 当 agent 表现不稳定时，问题未必在模型本身，也可能在工具边界、命名或输出结构。
