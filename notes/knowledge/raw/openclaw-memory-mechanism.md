# OpenClaw 记忆机制详解

**来源：** 技术文档 / 博客文章

## 两层记忆架构

### 1. 长期层：`MEMORY.md`
- 保存持久性事实、偏好和决策
- 跨会话长期保留
- 每次会话开始时加载

### 2. 短期层：`memory/YYYY-MM-DD.md`
- 每日日志文件
- 记录当日上下文和观察
- 自动加载今天和昨天的文件

---

## 记忆写入的三种方式

### 方式 1：用户明确告知（最可靠）
用户直接告诉系统记住某事。例如：
> "Add to USER.md that I want short answers and copy-pastable commands."

这是最可靠的写入方式。

### 方式 2：自动 flush（会话压缩前）
- 当会话上下文窗口接近上限时，触发内部静默写入
- 模型被提示将重要内容写入磁盘
- **问题**：写什么完全取决于模型自己决定，不保证所有有用信息都能落盘
- 结果：选择性的、不一致的长期记忆

### 方式 3："做梦" - 可选的后台整合（默认关闭）
- 收集短期信号，评分候选内容
- 只有通过评分、回忆频率和查询多样性门槛的内容才会晋升到长期记忆
- 阶段总结写入 `DREAMS.md` 供人工审查
- **注意**：需手动开启，默认禁用

---

## 记忆检索方式

| 工具 | 功能 |
| :--- | :--- |
| `memory_search` | 语义搜索，即使措辞与原文不同也能找到相关笔记 |
| `memory_get` | 读取特定记忆文件或行范围 |

**混合检索策略：** 结合向量相似度（语义匹配）+ BM25 关键词相关性（精确匹配 ID、代码符号等）

---

## 关键局限

1. **遗漏风险**：重要事实可能因模型忘记写入而丢失
2. **质量依赖**：记忆质量取决于 LLM 遵循持久化指令的能力
3. **检索挑战**：当日志积累过多时，需要智能体在正确的时间用正确的查询搜索 - 而模型在这方面表现不一致

---

## 核心结论

- **自动写入**（flush、做梦）存在但不可靠
- **最可靠的方式**：明确告诉系统要记住什么
- **设计优势**：文件是纯 Markdown，完全透明且可人工编辑
